Canonical URL Examples

Each example below shows a real-world scenario where canonical URLs are essential.

Example 1: Self-Referencing Canonical

Every page should include a canonical tag pointing to itself. This is the baseline implementation.

Current page URL:

https://example.com/blog/canonical-urls

Canonical tag in <head>:

<link rel="canonical" href="https://example.com/blog/canonical-urls" />
<head>
  <title>Canonical URLs Guide</title>
  <link rel="canonical" href="https://example.com/blog/canonical-urls" />
</head>
Why it matters: Self-referencing canonicals prevent dilution from tracking parameters (e.g., ?utm_source=twitter) and ensure search engines always know which URL is the authoritative version.

Example 2: WWW vs. Non-WWW Canonicalization

Choose one domain version (www or non-www) and consistently canonicalize to it.

Scenario: The site is accessible via both:

https://www.example.com/page
https://example.com/page

Preferred (with canonical tag on www version):

<link rel="canonical" href="https://example.com/page" />
<!-- On https://www.example.com/page -->
<link rel="canonical" href="https://example.com/page" />

<!-- On https://example.com/page (self-referencing) -->
<link rel="canonical" href="https://example.com/page" />
Best Practice: Also implement a 301 redirect from www to non-www (or vice versa) at the server level. The canonical tag serves as a backup signal, while the redirect provides the definitive user experience.

Example 3: Filtering and Sorting Parameters

E-commerce and listing pages often have multiple URLs for the same content due to filters and sorting.

Multiple URLs for the same product listing:

https://example.com/shoes
https://example.com/shoes?color=black
https://example.com/shoes?sort=price
https://example.com/shoes?page=2

Each filtered page should canonical to the master category page:

<link rel="canonical" href="https://example.com/shoes" />
<!-- On https://example.com/shoes?color=black -->
<link rel="canonical" href="https://example.com/shoes" />

<!-- On https://example.com/shoes?sort=price -->
<link rel="canonical" href="https://example.com/shoes" />
Important: For paginated content (page=2, page=3), consider using rel="next" and rel="prev" in addition to canonical. Some SEO experts recommend canonicalizing each paginated page to itself to preserve their individual value.

Example 4: HTTP vs. HTTPS Canonicalization

Always canonicalize to the HTTPS version for security and SEO best practices.

If both HTTP and HTTPS are accessible (shouldn't happen, but legacy sites):

http://example.com/page
https://example.com/page

Canonical to HTTPS:

<link rel="canonical" href="https://example.com/page" />
<!-- On http://example.com/page -->
<link rel="canonical" href="https://example.com/page" />

<!-- On https://example.com/page -->
<link rel="canonical" href="https://example.com/page" />
Note: In practice, you should implement HSTS (HTTP Strict Transport Security) and 301 redirect all HTTP traffic to HTTPS. The canonical tag is an additional safety measure.

Example 5: Trailing Slash Consistency

Choose whether your URLs end with a trailing slash or not, and canonicalize accordingly.

Both may be accessible depending on server configuration:

https://example.com/about
https://example.com/about/

Pick one and canonicalize:

<link rel="canonical" href="https://example.com/about/" />
<!-- On https://example.com/about (no trailing slash) -->
<link rel="canonical" href="https://example.com/about/" />

<!-- On https://example.com/about/ (with trailing slash) -->
<link rel="canonical" href="https://example.com/about/" />

Example 6: Cross-Domain Canonical for Syndicated Content

When content is republished on another site, use cross-domain canonicals to credit the original source.

Original article on Site A:

https://original-site.com/original-article

Syndicated copy on Site B:

https://syndicator-site.com/blog/original-article

Site B's canonical points to Site A:

<link rel="canonical" href="https://original-site.com/original-article" />
<!-- On syndicated copy at https://syndicator-site.com/blog/original-article -->
<link rel="canonical" href="https://original-site.com/original-article" />
Why: Cross-domain canonicals ensure that the original publisher receives the SEO credit for the content, even though it appears on other sites. This is standard practice for guest posts and syndicated content.

Example 7: Session ID Parameters

E-commerce and membership sites often append session IDs to URLs, creating many duplicate URLs.

Each user visit generates a unique URL:

https://example.com/products?sid=a1b2c3
https://example.com/products?sid=d4e5f6
https://example.com/products?sid=g7h8i9

All canonicalize to the base URL:

<link rel="canonical" href="https://example.com/products" />
<!-- On https://example.com/products?sid=a1b2c3 -->
<link rel="canonical" href="https://example.com/products" />

<!-- On https://example.com/products?sid=d4e5f6 -->
<link rel="canonical" href="https://example.com/products" />
Best Practice: Avoid session IDs in URLs altogether. Use cookies or server-side sessions instead. If you must use URL-based sessions, always include a self-referencing canonical without the session ID.

Example 8: Printer-Friendly Versions

Printer-friendly versions of pages should canonicalize to the standard version to avoid duplicate content.

Standard and print versions:

https://example.com/article
https://example.com/article?print=1

Print version canonicals to standard:

<link rel="canonical" href="https://example.com/article" />
<!-- On https://example.com/article?print=1 -->
<link rel="canonical" href="https://example.com/article" />

<!-- On https://example.com/article -->
<link rel="canonical" href="https://example.com/article" />
Better approach: Use CSS @media print for printer-friendly styling instead of creating a separate URL. This eliminates the duplicate content issue entirely.

Example 9: HTTP Header Canonical for PDFs

For non-HTML resources like PDFs, use HTTP response headers to specify the canonical URL.

PDF served at multiple URLs:

https://example.com/downloads/report.pdf
https://example.com/report.pdf?version=2

HTTP Response Header:

Link: <https://example.com/downloads/report.pdf>; rel="canonical"
HTTP/1.1 200 OK
Content-Type: application/pdf
Link: <https://example.com/downloads/report.pdf>; rel="canonical"
Implementation: Configure this at the server level (Apache .htaccess, Nginx config, or in your application framework's response handling).

Example 10: Canonical Validation Checklist

Use this checklist to verify your canonical implementation:

  1. Check for single canonical tag - Only one <link rel="canonical"> per page
  2. Verify absolute URL - Must include protocol and domain (https://example.com/page)
  3. Test URL resolves - Canonical URL must return 200 (not 404 or redirect)
  4. Match content - Canonical page content should be identical or very similar
  5. No canonical chain - A → B → C is invalid; should be A → C, B → C
  6. Consistent protocol - Use HTTPS consistently
  7. No conflicting signals - Noindex + canonical sends mixed signals
<!-- VALID canonical implementation -->
<link rel="canonical" href="https://example.com/preferred-page" />

<!-- INVALID: relative URL -->
<link rel="canonical" href="/preferred-page" />

<!-- INVALID: trailing slash mismatch without redirect -->
<link rel="canonical" href="https://example.com/preferred-page" />
<!-- ...but this URL redirects to https://example.com/preferred-page/ -->

<!-- INVALID: chain -->
<!-- Page A canonical points to Page B -->
<!-- Page B canonical points to Page C -->
