# Course DNA

## Student Evolution Framework

The curriculum transforms students through 6 distinct stages. Every module must move the student forward. Every module must contribute to the long-term journey.

### Stage 1: Complete Beginner (Modules 01–27)
**What they know:** Nothing. They have never written a line of code.

**What they can do after:** Write basic Python scripts using variables, data types, operators, strings, booleans, lists, tuples, sets, dictionaries. Understand control flow.

**Teaching approach:**
- Zero assumptions. Explain everything from first principles.
- Heavy use of analogies and mental models.
- Every concept connected to something the student already understands from daily life.
- Maximum hand-holding. Show exact keystrokes.
- No "obviously" or "as you can see."

### Stage 2: Comfortable Python User (Modules 28–55)
**What they know:** Basic syntax, data structures, control flow.

**What they can do after:** Write multi-file Python programs using functions, modules, packages, iterators, generators. Understand scope, recursion, lambda functions.

**Teaching approach:**
- Start removing training wheels gradually.
- Introduce professional tooling (pip, virtual environments).
- Show how concepts connect to build larger programs.
- Begin introducing "Pythonic" patterns vs "just works" patterns.

### Stage 3: Problem Solver (Modules 56–87)
**What they know:** Functions, modules, error handling, OOP fundamentals.

**What they can do after:** Design and implement class-based solutions. Handle files (text, binary, CSV, JSON). Use OS module, pathlib, temporary files. Write clean, maintainable code.

**Teaching approach:**
- Shift from "how do I write this" to "what is the best design for this."
- Introduce tradeoffs explicitly (readability vs performance, simplicity vs flexibility).
- Teach debugging as a systematic skill, not guesswork.
- Projects become multi-file with real-world requirements.

### Stage 4: Software Developer (Modules 88–142)
**What they know:** OOP, file handling, exception handling, basic modules.

**What they can do after:** Build database-backed applications (MySQL, MongoDB). Perform data analysis with NumPy, Pandas, SciPy. Create visualizations with Matplotlib. Handle real-world data pipelines.

**Teaching approach:**
- Shift to "production-ready" mindset.
- Introduce performance considerations, memory management, optimization.
- Projects involve data processing, cleaning, analysis.
- Teach database design principles alongside CRUD operations.

### Stage 5: Professional Engineer (Modules 143–196)
**What they know:** Data analysis, databases, visualization.

**What they can do after:** Implement machine learning models. Design and analyze data structures and algorithms. Build Django web applications with authentication. Write REST APIs. Apply clean code practices, debugging, logging, testing, and performance optimization.

**Teaching approach:**
- Everything is production code. No toy examples.
- Teach complexity analysis (Big O) alongside implementation.
- Introduce testing as a design practice, not an afterthought.
- Security-conscious development.
- Code review mindset — teach students to critique their own code.

### Stage 6: Advanced Python Specialist (Modules 197–206)
**What they know:** Everything from ML to web development to DSA to professional practices.

**What they can do after:** Build complete, real-world applications from scratch using Python-native UIs (tkinter, Flask, rich CLI). Integrate multiple technologies in a single project. Ship production-quality software.

**Teaching approach:**
- Full-stack project mindset.
- All projects include troubleshooting guides, test suites, and Python-native UI code.
- Emphasis on architecture, maintainability, and extensibility.
- Prepare students for technical interviews with real company-style questions.

## Curriculum Connectivity Rules

- Never teach concepts in isolation. Always connect concepts to previous learning.
- When introducing a concept, reference the module where the prerequisite was taught.
- Avoid unnecessary duplication of examples, projects, and interview questions across modules (tracked in knowledge graph).
- Each module should leave the student with a CLEAR sense of "where I was, where I am now, and where I'm going next."

## Course Identity

- Voice: Authoritative but warm. Like a brilliant senior engineer who genuinely enjoys teaching.
- Tone: Direct, clear, no fluff. Uses analogies freely but never at the expense of precision.
- Philosophy: Depth without fluff. First principles. WHY before HOW.
- Style: Every lesson reads like a conversation with a mentor, not a textbook.
