CSS Comments - Examples

Example 1: Section Header Comments

Use comments to organize stylesheets into logical sections.

/* ===== RESET & BASE ===== */
No margin, no padding, border-box on everything.
/* ===== TYPOGRAPHY ===== */
Body font: system-ui stack, headings use serif.
/* ===== LAYOUT ===== */
Container max-width 1200px, centered via auto margins.
/* ===== 1. RESET & BASE ===== */
*, *::before, *::after { box-sizing: border-box; margin: 0; }

/* ===== 2. TYPOGRAPHY ===== */
body { font-family: system-ui, sans-serif; line-height: 1.6; }
h1, h2 { font-family: Georgia, serif; }

/* ===== 3. LAYOUT ===== */
.container { max-width: 1200px; margin: 0 auto; padding: 1rem; }

Example 2: Good Comments (Explain WHY)

Good comments explain the reasoning behind a choice, not what the code does.

Use flexbox for equal-height cards in the grid

Card 1

All cards match height.

Card 2

Even with different content amounts.

Respect user motion preferences
In a real browser with prefers-reduced-motion: reduce, all animations are disabled.
/* Use flexbox for equal-height cards in grid */
.card-grid { display: flex; flex-wrap: wrap; }

/* Respect reduced motion — a11y requirement */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
  }
}

Example 3: Bad Comments (Explain WHAT — redundant)

Avoid comments that merely restate the code. They add noise without value.

/* This sets the color to blue */ — your code already says `color: blue`
/* This adds 20px of padding */ — the property name is `padding: 20px`
Don't state the obvious. Explain WHY, not WHAT.
/* ❌ Redundant — code is self-documenting */
h1 { color: blue; }  /* This sets color to blue */
.card { padding: 20px; }  /* This adds 20px of padding */

/* ✅ Good — explains the reasoning */
h1 { color: #1a365d; }  /* Brand navy — matches company logo */
.card { padding: 20px; }  /* Matches grid gutter in design system */

Example 4: Disabling Code with Comments

Wrap code in /* ... */ to temporarily disable it during development.

This section is active and renders normally.
Commented-out CSS is not applied.
/* Temporarily disabled — causes layout shift on mobile
.featured-card {
  transform: scale(1.05);
  transition: transform 0.3s ease;
}
*/

/* Active */
.card { background: white; border-radius: 8px; }

Example 5: TODO, FIXME, and HACK Comments

Standardized prefixes make it easy to search for technical debt.

TODO: Replace hardcoded colors with CSS custom properties
FIXME: Border-radius clipped in Safari < 14
HACK: Negative margin compensates for float clearing issue
OPTIMIZE: Rebuild this layout with CSS Grid to reduce DOM nodes
/* TODO: Replace hardcoded colors with CSS custom properties */
/* FIXME: Border-radius clips on Safari < 14 */
/* HACK: Negative margin offsets float clearing gap */
/* OPTIMIZE: Convert this flex layout to CSS Grid */

Example 6: Inline Comments

End-of-line comments document a single property's rationale.

.banner {
  background: #0f172a; /* Dark bg matching brand guidelines */
  padding: 3rem 2rem;
  font-size: 1.5rem; /* 24px / 16px base */
  color: #f8fafc; /* High contrast for readability */
} 
.banner {
  background: #0f172a;   /* Brand dark — matches company palette */
  padding: 3rem 2rem;
  font-size: 1.5rem;     /* 24px computed from 16px base */
  color: #f8fafc;        /* WCAG AAA contrast on dark bg */
}

Example 7: Multi-Line Block Comments for Documentation

Block comments can document entire components or modules.

Component: UserAvatar
Usage: <div class="avatar"><img src="..."></div>
States: default, .avatar--online, .avatar--offline 
/**
 * UserAvatar Component
 *
 * Renders a circular user avatar with optional online status.
 * Usage:
 *   <div class="avatar">
 *     <img src="user.jpg" alt="User">
 *   </div>
 *
 * Modifiers:
 *   .avatar--online  — green status indicator
 *   .avatar--offline — gray status indicator
 */
.avatar {
  width: 40px; height: 40px;
  border-radius: 50%;
  overflow: hidden;
}
